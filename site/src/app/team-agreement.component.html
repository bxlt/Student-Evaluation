Version Control

Descriptive Commit Message
Commit message headers should be verbose, describing what/where the changes are/were made. If additional details are needed, it should go in the body.
Commit message header should not exceed 50 characters
Additional information should be in the body.
Master branch must always build and run.
Any branches should be well tested (i.e., with sufficient test cases) before merging to the Master branch. If there is a bug on the Master branch, the team will evaluate the bug. If it's a major bug, the Master branch will be rebased to the a previous commit. Otherwise, the bug will be fixed promptly.
A new branch will be created for each feature/release
Branches will be locked after feature/release has been merged to Master.
Auto-generated files should not be commited
These files should be properly removed in .gitignore.
ZenHub for project tracking
All features should be a task in ZenHub. Commits will be associated to a feature.

Contingency Planning

To try to ensure the project goes as smoothly as possible, we want to plan out actions for some emergency cases.
Emergency Situation Action
Team Member unavailable for a period of time.
(E.g. sick, busy)
Team member should let the rest of the team be aware of their unavailability ASAP
Schedule emergency meeting (within 1 day) so the team can re-distribute the tasks for the week
Speak to TA if several members are unavailable for a sprint to see if tasks can be reduced
Team Member unavailable permanently
(E.g. dropped course)
Team member should let the rest of the team be aware of their unavailability ASAP
Schedule emergency meeting (within 1 day) so the team can re-distribute the tasks for the week
Make sure TA is aware that a team member is permanently unavailable
Re-access the project to see if it could still be completed on time or if features need to be dropped
Team member is academically dishonest
All team members need to review the feature from a branch before merging the branch to production
During code reviews, if academic dishonesty is found in a feature, Master branch needs to be rebased to a previous commit when the feature wasn't present or the feature needs to be removed ASAP.
